# Save — 存档管理器（v0.2 Delta）

> 来源: add-game-cluster-save | 状态: 草案 | 变更类型: ADDED

## 概述

新增 Save 模块，实现游戏存档管理器。支持定时存档（可配置间隔）、手动存档接口（关键操作触发）和事件触发存档（玩家下线、切换场景）。存档数据通过 DBProxy 写入 MongoDB。DBProxy(主) 在自身内存中维护 Game A 的状态备份，Game A 挂了时 DBProxy(主) 自动将备份数据存档到 MongoDB（由 DBProxy 模块负责，不在 Save 模块范围内）。通过存档队列避免并发写冲突。

---

## ADDED Requirements

### Requirement: 定时存档

Save 模块 SHALL 支持可配置间隔的定时存档，将游戏状态定期持久化到数据库。

定时存档 MUST 满足以下要求：
- 存档间隔可配置，默认 5 分钟
- 每次定时存档 MUST 在帧边界执行，确保状态一致性
- 定时存档 MUST 通过存档队列提交，避免与手动存档冲突
- 定时器 MUST 在 Game A 进程中运行

#### Scenario: 默认 5 分钟定时存档

- **WHEN** Game A 进程运行超过 5 分钟且未手动修改存档间隔
- **THEN** 系统自动触发一次定时存档
- **AND** 存档包含当前帧的完整游戏状态
- **AND** 存档完成后记录 INFO 级别日志，包含存档耗时和数据大小

#### Scenario: 自定义存档间隔

- **WHEN** 配置存档间隔为 10 分钟
- **THEN** 系统每 10 分钟触发一次定时存档
- **AND** 两次存档之间不触发额外的定时存档

#### Scenario: 定时存档与帧边界

- **WHEN** 定时器在帧 N 执行过程中到期
- **THEN** 存档操作延迟到帧 N 执行完毕后触发
- **AND** 存档的快照状态与帧 N 结束时的状态一致
- **AND** 帧 N+1 在存档完成前不开始执行

### Requirement: 手动存档接口

Save 模块 SHALL 提供手动存档接口，允许关键操作触发即时存档。

手动存档接口 MUST 支持以下调用方式：
- Lua 脚本层调用 `save_trigger(reason)` 触发存档
- C 内核层调用 `ce_save_trigger(reason)` 触发存档
- 手动存档 MUST 通过存档队列提交，与定时存档共享队列
- 手动存档优先级 MUST 高于定时存档（队列中优先执行）

#### Scenario: 关键操作触发手动存档

- **WHEN** 玩家完成重要任务（如击败 Boss、完成交易）
- **THEN** Lua 脚本调用 `save_trigger("boss_defeated")` 触发存档
- **AND** 存档请求进入队列，优先于等待中的定时存档执行
- **AND** 存档完成后回调通知 Lua 层

#### Scenario: GM 命令触发存档

- **WHEN** 管理员执行 GM 命令 `/save`
- **THEN** 系统调用 `ce_save_trigger("gm_command")` 触发即时存档
- **AND** 存档完成后向管理员返回确认消息，包含存档 ID 和时间戳

#### Scenario: 手动存档失败处理

- **WHEN** 手动存档因数据库连接失败而无法完成
- **THEN** 系统重试最多 3 次，间隔 1s
- **AND** 3 次重试均失败后记录 ERROR 日志
- **AND** 向调用方返回失败状态和错误原因

---

### Requirement: 事件触发存档

Save 模块 SHALL 支持基于游戏事件自动触发存档。

事件触发存档 MUST 覆盖以下场景：
- 玩家下线（`player_logout`）：玩家主动或被动断开连接时存档
- 玩家切换场景（`player_change_scene`）：玩家从一个场景传送到另一个场景时存档
- 服务器关闭（`server_shutdown`）：服务器正常关闭前存档所有在线玩家

事件触发存档 MUST 通过存档队列提交，与定时存档和手动存档共享队列。

#### Scenario: 玩家下线时存档

- **WHEN** 玩家主动退出游戏或连接超时断开
- **THEN** 系统触发 `player_logout` 事件存档
- **AND** 存档包含该玩家的完整状态（位置、背包、任务进度等）
- **AND** 存档完成后玩家数据从内存中安全移除

#### Scenario: 玩家切换场景时存档

- **WHEN** 玩家通过传送门从场景 A 切换到场景 B
- **THEN** 系统在玩家离开场景 A 时触发 `player_change_scene` 事件存档
- **AND** 存档保存玩家在场景 A 的最终状态
- **AND** 玩家在场景 B 加载完成后，场景 B 的状态基于最新存档恢复

#### Scenario: 服务器正常关闭时存档

- **WHEN** 服务器收到 SIGTERM 信号准备正常关闭
- **THEN** 系统触发 `server_shutdown` 事件
- **AND** 所有在线玩家的状态依次存档
- **AND** 所有存档完成后服务器进程退出
- **AND** 记录每个玩家存档的成功/失败状态

---

### Requirement: 存档队列

Save 模块 SHALL 使用存档队列管理并发存档请求，确保同一时刻只有一个存档操作在执行。

存档队列 MUST 满足以下要求：
- 所有存档请求（定时、手动、事件触发）统一进入 FIFO 队列
- 队列 MUST 支持优先级：手动存档 > 事件触发存档 > 定时存档
- 同一玩家的多次存档请求 MUST 合并（仅保留最新请求）
- 队列深度 MUST 有上限（默认 256），超出时丢弃最低优先级的旧请求

#### Scenario: 并发存档请求排队

- **WHEN** 定时存档正在执行时，同时收到手动存档和玩家下线存档请求
- **THEN** 手动存档和下线存档请求进入队列等待
- **AND** 当前定时存档完成后，按优先级执行：手动存档优先于下线存档
- **AND** 所有请求依次执行，无并发写冲突

#### Scenario: 同一玩家请求合并

- **WHEN** 玩家 A 的存档请求已在队列中等待，又收到玩家 A 的新存档请求
- **THEN** 队列中玩家 A 的旧请求被新请求替换
- **AND** 仅保留最新的存档请求，避免重复存档

#### Scenario: 队列溢出保护

- **WHEN** 存档队列已达到最大深度 256
- **THEN** 新的低优先级存档请求（定时存档）被丢弃
- **AND** 高优先级请求（手动存档、事件触发）仍可入队（替换最低优先级的定时存档）
- **AND** 记录 WARN 日志，包含丢弃的请求信息
