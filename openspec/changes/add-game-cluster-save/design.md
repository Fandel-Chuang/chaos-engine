# Design: Game 实时备份 + DBProxy 主备 + 存档系统

## Context

ChaosEngine v0.2 已具备以下能力：

- **AOI 十字链表**（`ce_aoi.h/c`）：实体进出视野、移动检测、周围实体查询，纯 C99 单线程安全。
- **Cell 大地图管理**（`ce_cell.h/c`）：世界分块、动态分裂/合并、跨 Cell 查询、进程分配。
- **Admin IPC**（`ce_admin_ipc.h/c`）：JSON-RPC 2.0 over Unix Domain Socket，独立线程，支持运行时查询。
- **网络层**：TCP Echo Server（端口 7777），io_uring 异步 I/O 后端，eBPF 可观测性。
- **Lua Admin 进程**：Lapis Web 管理后台，独立进程，通过 IPC 查询 server 状态。

**当前缺失**：

1. **无集群支持**：单进程运行，宕机即服务中断，无法水平扩展。
2. **无进程间同步**：Cell 已支持 `process_id` 分配字段，但无实际的多进程通信机制。
3. **无数据持久化**：所有游戏状态仅存于内存，重启全部丢失。
4. **无数据库层**：玩家数据、装备、背包等无存储和查询能力。

本设计为 v0.3 引入 **Game 实时备份** + **DBProxy 主备** + **定时/手动存档** 三大子系统，使 ChaosEngine 从单机原型进化为可生产部署的分布式游戏服务器。

## Goals / Non-Goals

**Goals:**

1. **进程间状态同步**：基于 C 内核的 `ce_sync` 模块，支持 Game → DBProxy 单向状态推送。主 Game 进程（A）将每帧 ECS 状态变更通过 TCP 推送给 DBProxy(主)，DBProxy 在内存中维护状态镜像（实时备份）。Game 之间无直接 TCP 连接。
2. **故障处理**：Game A 挂了 → 玩家掉线 → DBProxy(主) 将内存中的备份数据存档到 MongoDB → 玩家重新登录时从 MongoDB 恢复；DBProxy(主) 挂了 → Game A 切到本机 DBProxy(备)。
3. **DBProxy 主备**：Lua 实现的 DBProxy 进程，封装 MongoDB 操作 + Game 内存状态实时备份，支持主备自动切换。主 DBProxy 在远程机器，本机为备用。Game 进程通过 C 客户端（`ce_dbproxy`）与之通信。
4. **存档系统**：`ce_save` 模块支持定时存档（可配置间隔）+ 手动存档（Admin 命令触发），存档数据经 DBProxy 写入 MongoDB。
5. **玩家登录加载**：玩家登录时从 DBProxy 加载持久化数据，恢复到 Game 进程内存。
6. **二进制协议**：Game↔DBProxy 之间使用自定义二进制协议（长度前缀 + 消息类型 + protobuf-like 变长编码），低延迟高吞吐。
7. **心跳与故障检测**：Game↔DBProxy 之间心跳检测，DBProxy 主备间心跳检测，超时自动切换。

**Non-Goals:**

- ❌ 不做跨服（Cross-Server）通信（不同 Game 集群之间的玩家交互延后到 v0.4）。
- ❌ 不做 MongoDB 分片集群（Sharded Cluster）—— 当前使用单机 MongoDB + 副本集即可。
- ❌ 不做全自动存档（每次操作都写库）—— 性能开销过大，采用定时 + 手动模式。
- ❌ 不做玩家数据迁移工具（延后）。
- ❌ 不做 DBProxy 的 SQL 数据库支持（仅 MongoDB）。
- ❌ 不做 Game 进程的动态扩缩容（Cell 迁移到新进程延后）。

## Decisions

### Decision 1: Game → DBProxy 状态同步 vs Game ↔ Game 直连同步

**选择**：Game → DBProxy 单向状态推送，DBProxy 存 Game 内存实时备份。

**理由**：
- **解耦 Game 与数据库**：Game A 只与 DBProxy 通信，不直连 MongoDB。DBProxy 中间层隔离 Game 与数据库，简化连接管理。
- **集中式实时备份**：DBProxy(主) 在自身内存中维护 Game A 状态的完整镜像，作为权威的实时备份数据源。Game A 挂了时，DBProxy(主) 自动将备份数据存档到 MongoDB，玩家重新登录时从 MongoDB 恢复。
- **带宽效率**：Game A 每帧将增量变更（dirty entities 或操作指令）推送给 DBProxy，通常只有几十到几百字节。DBProxy 按帧序应用变更，保持内存镜像一致。
- **故障恢复简单**：Game A 挂了 → 玩家掉线 → DBProxy(主) 存档到 MongoDB，数据不丢失；DBProxy(主) 挂了 → Game A 切到本机 DBProxy(备)，数据库服务不中断。
- **调试友好**：DBProxy 中的状态镜像是可查询的，便于排查同步问题。

**替代方案**：
- ❌ Game ↔ Game 直连同步：Game 进程间需要维护 TCP 连接和心跳，增加了连接管理的复杂度；故障恢复时需要操作日志追赶（catch-up），增加恢复延迟。
- ❌ 主备共享内存：仅限单机，无法扩展到多机部署；共享内存 corruption 会同时影响主备。

**实现细节**：
- 同步协议：Game A → DBProxy 单向推送 SyncFrame（帧序号 + 变更实体列表）。
- DBProxy 按帧序号顺序应用变更到内存镜像。
- Game A 挂了时，玩家掉线，DBProxy(主) 将内存镜像序列化后存档到 MongoDB。

### Decision 2: DBProxy 主备 vs 直连 MongoDB

**选择**：DBProxy 主备代理模式 + Game 内存状态实时备份。

**理由**：
- **连接池复用**：多个 Game 进程共享 DBProxy 的 MongoDB 连接池，避免每个 Game 进程独立维护数十个数据库连接。
- **故障切换透明**：MongoDB 主从切换时，只需 DBProxy 内部处理重连，Game 进程无感知。
- **协议抽象**：Game 进程使用简单的二进制协议（save/load/query/sync），不依赖 MongoDB 驱动，降低 C 内核的依赖复杂度。
- **安全边界**：DBProxy 可以实施访问控制、请求限流、操作审计，Game 进程不直接暴露数据库凭证。
- **独立扩展**：DBProxy 可以独立扩容（增加进程数），不影响 Game 进程。
- **Game 内存状态实时备份**：DBProxy(主) 在自身内存中维护 Game A 状态的完整镜像，作为实时备份数据源。Game A 挂了时，玩家掉线，DBProxy(主) 自动将备份数据存档到 MongoDB。

**替代方案**：
- ❌ 直连 MongoDB：每个 Game 进程需要嵌入 MongoDB C 驱动（mongoc），增加 C 内核依赖和编译复杂度；连接数膨胀；故障切换逻辑需要在每个 Game 进程中实现。
- ❌ HTTP REST 网关：增加 HTTP 协议开销，延迟高（~1ms vs ~100μs），不适合高频存档操作。

**DBProxy 主备切换流程**：
1. 主 DBProxy 定期向备 DBProxy 发送心跳（TCP）。
2. 备 DBProxy 在 3 个心跳周期内未收到心跳，判定主宕机。
3. Game A 的 `ce_dbproxy` 客户端自动切换到本机备 DBProxy 地址。
4. 本机备 DBProxy 承担数据库代理职责（不存储 Game A 内存备份数据）。

### Decision 3: 二进制协议 vs JSON-RPC

**选择**：自定义二进制协议（长度前缀 + 消息类型 + 变长编码字段）。

**理由**：
- **性能**：二进制编解码比 JSON 解析快 5-10x，对于高频存档操作（每秒数百次）至关重要。
- **带宽**：二进制协议比 JSON 紧凑 3-5x，例如整数 123456 在 JSON 中占 6 字节，变长编码仅需 3 字节。
- **类型安全**：二进制协议有明确的消息类型枚举和字段定义，编译期即可检查，JSON 的字符串 key 容易拼写错误。
- **零依赖**：手写编解码器 ~500 行 C 代码，不依赖 protobuf 等第三方库。

**替代方案**：
- ❌ JSON-RPC（Admin IPC 使用的协议）：人类可读适合调试，但编解码开销大，不适合高频数据通路。
- ❌ Protocol Buffers：需要 protoc 编译步骤，增加构建依赖；生成的 C 代码量大；对于简单 save/load 操作过度设计。
- ❌ MessagePack：需要第三方库，且不如自定义协议可控。

**协议格式**：
```
消息格式:
┌──────────────┬──────────────┬──────────────────────────┐
│  length (4B) │  type  (2B)  │  payload (变长)          │
│  大端序       │  大端序       │                          │
└──────────────┴──────────────┴──────────────────────────┘

消息类型:
  0x0001  DB_SAVE_PLAYER    保存玩家数据
  0x0002  DB_LOAD_PLAYER    加载玩家数据
  0x0003  DB_SAVE_WORLD     保存世界状态
  0x0004  DB_LOAD_WORLD     加载世界状态
  0x0005  DB_QUERY           通用查询
  0x0006  DB_HEARTBEAT       心跳
  0x0007  DB_MASTER_CHANGE   主备切换通知
  0xFFFF  DB_ERROR           错误响应

变长整数编码: 每字节最高位为继续标志，低 7 位为数据。
字符串: 变长长度前缀 + UTF-8 数据。
```

### Decision 4: 定时存档 + 手动 vs 全自动

**选择**：定时存档（可配置间隔，默认 5 分钟）+ 手动触发（Admin 命令）。

**理由**：
- **性能可控**：定时存档将 I/O 操作集中到固定时间点，避免每次操作都写库导致的延迟抖动。
- **数据安全**：5 分钟间隔意味着最多丢失 5 分钟数据，对于大多数游戏场景可接受。
- **灵活性**：关键事件（Boss 战结束、玩家下线、服务器维护前）可通过 Admin 命令手动触发即时存档。
- **批量优化**：定时存档可以将多个玩家的数据批量写入，利用 MongoDB 的批量插入（bulkWrite）提升吞吐。

**替代方案**：
- ❌ 全自动（每次操作写库）：延迟不可控，数据库成为瓶颈；高频写操作（如玩家移动）会产生海量 I/O。
- ❌ 仅手动存档：运维负担重，容易遗忘；宕机时可能丢失数小时数据。

**存档策略**：
- **全量存档**：每 N 次定时存档执行一次全量（默认每 6 次，即 30 分钟），其余为增量。
- **增量存档**：仅保存自上次存档以来变更的实体数据（dirty flag 机制）。
- **存档触发条件**：定时器到期、玩家下线、Admin 命令、进程优雅关闭（SIGTERM）。

### Decision 5: Lua DBProxy vs C DBProxy

**选择**：Lua 实现 DBProxy。

**理由**：
- **快速迭代**：数据库代理层逻辑（连接池、重试、主备切换）变化频繁，Lua 的动态特性适合快速迭代。
- **MongoDB 生态**：Lua 有成熟的 MongoDB 驱动（如 lua-mongo、mongorover），C 语言的 MongoDB 驱动（mongoc）编译复杂、依赖多。
- **与 Admin 一致**：Admin 进程已使用 Lua + Lapis，DBProxy 使用相同技术栈，降低运维复杂度。
- **协程并发**：Lua 协程（Lapis cqueues 模式）天然适合处理大量并发数据库请求，无需多线程。
- **热更新**：Lua 代码可以热加载，DBProxy 逻辑更新无需重启进程。

**替代方案**：
- ❌ C DBProxy：开发效率低，MongoDB C 驱动（mongoc 1.x）API 复杂，编译依赖多（libbson、libmongoc、OpenSSL 等）；修改连接池逻辑需要重新编译。
- ❌ Go DBProxy：引入新语言运行时，增加部署复杂度；与引擎 Lua 生态不一致。

**Lua DBProxy 内部结构**：
```
dbproxy/
├── init.lua              # 入口，启动 TCP 服务器
├── protocol.lua          # 二进制协议编解码
├── connection.lua        # Game 连接管理（连接池）
├── mongo_client.lua      # MongoDB 操作封装
├── heartbeat.lua         # 主备心跳逻辑
├── failover.lua          # 故障处理逻辑
├── save_handler.lua      # 存档请求处理
├── load_handler.lua      # 加载请求处理
└── config.lua            # 配置（端口、Mongo URI、心跳间隔）
```

### Decision 6: 心跳间隔和超时阈值

**选择**：心跳间隔 1 秒，超时阈值 3 秒（3 个心跳周期）。

**理由**：
- **1 秒心跳**：足够频繁以快速检测故障，又不会产生显著的网络开销（每条心跳 < 100 字节）。
- **3 秒超时**：容忍短暂的网络抖动（单次丢包），但不会让故障检测过于迟钝。对于游戏服务器，3 秒的切换时间在玩家可接受范围内（短暂卡顿后重连）。
- **与行业惯例一致**：大多数游戏服务器的故障检测时间为 3-10 秒，3 秒属于激进但合理的设置。

**替代方案**：
- ❌ 500ms 心跳 + 1.5s 超时：网络开销翻倍，误判风险增加（网络抖动可能导致频繁切换）。
- ❌ 5s 心跳 + 15s 超时：故障检测过慢，玩家等待时间过长。

**心跳机制细节**：
- Game → DBProxy：Game 每 1 秒发送 `SYNC_HEARTBEAT` 消息（含最后帧序号）。
- DBProxy → Game：DBProxy 收到心跳后回复 `SYNC_HEARTBEAT_ACK`。
- DBProxy 主→备：主 DBProxy 每 1 秒发送 `DB_HEARTBEAT` 消息。
- 超时处理：连续 3 次未收到响应，判定对端宕机，触发切换流程。

## Risks / Trade-offs

### Risk 1: 状态同步延迟导致 DBProxy 存档时状态落后

- **风险**：网络延迟或 Game A 崩溃瞬间未同步的最后一帧数据，导致 DBProxy 内存镜像略微落后于 Game A 实际状态，存档到 MongoDB 的数据可能缺少最后一帧变更。
- **缓解**：
  - 心跳消息携带最后帧序号，DBProxy 可检测同步进度。
  - DBProxy 存档前确保已应用所有已收到的 SyncFrame。
  - 同步使用 TCP 保证顺序和可靠传输。
- **影响**：低。最多丢失 Game A 崩溃前最后一帧（~16ms）的变更数据。

### Risk 2: DBProxy 主备切换时的短暂不可用

- **风险**：DBProxy(主) 故障到 Game A 切换到本机 DBProxy(备) 之间存在短暂窗口（3s 心跳超时），期间数据库操作不可用。
- **缓解**：
  - 切换时间控制在 3s 心跳超时内，对玩家影响有限。
  - 切换期间 Game A 的存档请求进入本地队列，切换完成后重试。
  - 本机 DBProxy(备) 始终保持待命，切换无需额外启动时间。
- **影响**：低。3s 的短暂不可用对游戏体验影响可控。

### Risk 3: DBProxy 单点性能瓶颈

- **风险**：所有 Game 进程的数据库请求都经过单个 DBProxy 进程，可能成为瓶颈。
- **缓解**：
  - Lua 协程模型天然支持高并发（数千并发请求）。
  - MongoDB 连接池复用减少连接开销。
  - 未来可扩展为多 DBProxy 分片（按 uid hash 分配）。
- **影响**：低。当前阶段仅 Game A 一个进程，单个 DBProxy 足够。

### Risk 4: 存档数据一致性

- **风险**：定时存档期间如果进程崩溃，可能丢失未保存的数据；增量存档的 dirty flag 可能遗漏某些变更。
- **缓解**：
  - 存档前先写入操作日志（WAL 模式），崩溃后可从日志恢复。
  - 关键操作（交易、装备变更）立即标记 dirty flag。
  - 定期全量存档作为增量存档的校验基准。
- **影响**：中。最多丢失一个存档间隔（5 分钟）的数据。

### Risk 5: 二进制协议版本兼容性

- **风险**：DBProxy 升级后协议格式变化，旧版本 Game 进程无法通信。
- **缓解**：
  - 协议头部包含版本号字段。
  - DBProxy 支持协议版本协商：连接建立时交换支持的最高版本。
  - 滚动升级：先升级 DBProxy（兼容旧协议），再升级 Game 进程。
- **影响**：低。初期版本迭代快，但协议设计预留了版本字段。

### Trade-off: Game → DBProxy 同步 vs Game ↔ Game 直连

- **选择**：Game → DBProxy 单向状态推送，DBProxy 存 Game 内存实时备份。
- **代价**：DBProxy 需要额外内存存储 Game 状态镜像；DBProxy 成为同步的关键路径，其性能影响同步延迟。
- **收益**：Game A 与数据库解耦；DBProxy(主) 集中维护实时备份，Game A 挂了自动存档到 MongoDB，玩家重登时数据不丢失；DBProxy 主备保证数据库服务不中断。
- **缓解**：DBProxy 使用高效内存数据结构存储状态镜像；同步使用 TCP 长连接减少握手开销。

### Trade-off: 定时存档间隔

- **选择**：5 分钟定时 + 手动触发。
- **代价**：最多丢失 5 分钟数据。
- **收益**：数据库写入压力可控（每秒约 0.003 次写入 per player vs 60 次全自动）。
- **可调**：间隔通过配置文件调整，生产环境可根据实际需求设为 1-10 分钟。
